# Offers System - Endpoints & Workflow Guide

> Base URL: `/api/offers`
> Offers are rewards that users **spend loyalty points** to redeem.
> Each offer has a **fixed points cost** and a **fixed monetary value** (what Cable pays the provider).
> Offer codes follow the format: `CBL-XXXXXX`

---

## How It Works (Real-World Flow)

```
 PROVIDER STAFF                ADMIN                        USER (Mobile App)
  |                              |                              |
  |  1. Propose new offer        |                              |
  |  (set title, description,   |                              |
  |   points cost, monetary     |                              |
  |   value, limits)            |                              |
  |---------------------------->|                              |
  |                              |                              |
  |                       2. Admin reviews                     |
  |                          and approves                      |
  |                          (or rejects)                      |
  |                              |                              |
  |                       3. Offer goes live                   |
  |                          (IsActive = true)                 |
  |                              |                              |
  |                              |          4. User sees offer  |
  |                              |             in active list   |
  |                              |             (100 pts = Free  |
  |                              |              Coffee, etc.)   |
  |                              |                              |
  |  5. Customer wants to        |                              |
  |     redeem the offer at      |                              |
  |     provider location        |                              |
  |                              |                              |
  |  6. Staff opens app,         |                              |
  |     selects the offer,       |                              |
  |     gets QR code (CBL-XXXXXX)|                              |
  |                              |                              |
  |  7. Staff shows QR to customer                              |
  |     -------------------------+----------------------------->|
  |                              |                              |
  |                              |          8. User opens app,  |
  |                              |             scans QR code    |
  |                              |                              |
  |                              |          9. System auto:     |
  |                              |             - Deducts points |
  |                              |             - Completes TX   |
  |                              |             - Shows result   |
  |                              |                              |
  |                       10. Monthly settlement               |
  |                           generated by admin               |
  |                           (offer + partner TXs)            |
  |                              |                              |
  |                       11. Admin marks settlement            |
  |                           as Invoiced -> Paid              |
  |                           (Cable pays provider)            |
  |                              |                              |
```

### Step-by-Step:

1. **Provider staff proposes an offer** through the app -- sets title (English + Arabic), description, points cost (e.g. 100 points), monetary value (e.g. 3.000 KWD that Cable pays provider), currency, max uses, code expiry time, and validity dates.

2. **Admin reviews the offer** and either approves or rejects it (with an optional note).

3. **Once approved**, the offer becomes active and visible to users (`IsActive = true`, `ApprovalStatus = Approved`).

4. **Users browse active offers** in the Cable app. Each offer shows its points cost (e.g. "Free Coffee - 100 points").

5. **Customer visits the partner location** and wants to redeem an offer using their loyalty points.

6. **Provider staff opens the Cable app**, selects the approved offer, and taps "Generate Code". The system creates a `CBL-XXXXXX` code with the offer's fixed points cost and monetary value snapshotted.

7. **Staff shows the QR code** (containing `CBL-XXXXXX`) to the customer.

8. **User opens Cable app**, goes to the QR scanner screen, and scans the code.

9. **System automatically**:
   - Finds the transaction by code
   - Checks if code has expired
   - Checks max uses per user for this offer
   - **Deducts loyalty points** from user's wallet (fails if insufficient balance)
   - Assigns the user to the transaction
   - Marks it as Completed
   - Returns a summary with offer title, points deducted, and monetary value

10. **At month-end**, admin runs settlement generation -- all completed offer and partner transactions are grouped by provider and aggregated into one settlement record per provider per month.

11. **Admin manages the settlement lifecycle** -- marks as Invoiced, then Paid (recording the actual paid amount), or Disputed if there's an issue. Cable pays the provider the total monetary value of redeemed offers.

## Offer Approval Status Values

| Status | Value | Meaning |
|--------|-------|---------|
| Pending | 1 | Offer proposed, waiting for admin review |
| Approved | 2 | Admin approved, offer is active |
| Rejected | 3 | Admin rejected (with optional note) |

---

## Transaction Status Values

| Status | Value | Meaning |
|--------|-------|---------|
| Initiated | 1 | Provider created the code, waiting for user to scan |
| Completed | 2 | User scanned the code, points deducted |
| Expired | 3 | Code expired before user scanned it |
| Cancelled | 4 | Provider cancelled the code before user scanned it |

---

## Settlement Status Values

| Status | Value | Meaning |
|--------|-------|---------|
| Pending | 1 | Settlement generated, waiting for admin review |
| Invoiced | 2 | Invoice created/sent to provider |
| Paid | 3 | Payment transferred to provider |
| Disputed | 4 | Provider disputed the settlement amount |

### 3. Scan Offer Code (QR Scan)

User scans the QR code shown by the provider staff. This is the main action -- it **deducts loyalty points** from the user and completes the transaction.

```
POST /api/offers/ScanOfferCode?code=CBL-AB3K9X
```

**Requires:** Authorization (JWT token)

**Query Parameters:**
| Param | Type | Required | Description |
|-------|------|----------|-------------|
| code | string | Yes | The `CBL-XXXXXX` code from the QR |

**What happens internally:**
1. Finds the transaction by code (must be `Initiated` status)
2. Checks if the code has expired -> if expired, marks as `Expired` and returns error
3. Checks max uses per user for this offer -> if exceeded, returns error
4. **Deducts points from user's wallet** (if insufficient balance, throws error and transaction stays `Initiated`)
5. Sets `UserId` to the current user
6. Marks transaction as `Completed`
7. Returns a summary with offer title, points deducted, and monetary value

**Response:** `200 OK`
```json
{
  "offerTitle": "Free Coffee Reward",
  "pointsDeducted": 100,
  "monetaryValue": 3.000,
  "currencyCode": "KWD"
}
```

**Errors:**
- `401` Unauthorized (not logged in)
- `404` Code not found or already used/completed
- `422` Code has expired
- `422` Maximum usage limit reached for this offer
- `422` Insufficient points balance

|-------|------|----------|-------------|
| title | string | Yes | Offer title (max 255 chars) |
| titleAr | string | No | Arabic title |
| description | string | No | English description |
| descriptionAr | string | No | Arabic description |
| providerType | string | Yes | `"ChargingPoint"` or `"ServiceProvider"` |

### 7. Create Transaction (Generate QR Code)

Provider staff creates a new transaction by selecting an offer. The system generates a `CBL-XXXXXX` code that the staff shows to the customer as a QR code. No amount entry needed -- the offer has fixed points cost and monetary value.

```
POST /api/offers/provider/CreateTransaction
```

**Requires:** Authorization (JWT token)

**Request Body:**
```json
{
  "offerId": 1
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| offerId | int | Yes | The approved offer ID |

**What happens internally:**
1. Loads the offer
2. Validates:
   - Offer is active, approved, within validity dates
   - Max total uses not exceeded
3. Snapshots from the offer:
   - `pointsDeducted` = offer's `pointsCost`
   - `monetaryValue` = offer's `monetaryValue`
   - `currencyCode` = offer's `currencyCode`
4. Generates a unique `CBL-XXXXXX` code
5. Sets `codeExpiresAt` based on offer's `offerCodeExpiryMinutes`
6. Stores `ConfirmedByUserId` = the staff member who created it
7. Does NOT set `UserId` yet (set when user scans)
8. Increments `offer.CurrentTotalUses++`

**Response:** `200 OK`
```json
{
  "offerCode": "CBL-AB3K9X",
  "expiresAt": "2025-06-15T10:30:00Z",
  "pointsCost": 100,
  "monetaryValue": 3.000,
  "currencyCode": "KWD"
}
```

**Errors:**
- `401` Unauthorized
- `404` Offer not found, not active, or not approved
- `422` Offer not yet available / has expired / max uses reached

### 19. Generate Settlement

Generate monthly settlements for all providers with completed transactions in the given month. This aggregates BOTH offer transactions AND partner transactions into one settlement per provider.

```
POST /api/offers/GenerateSettlement
```

**Requires:** Authorization (JWT token)

**Request Body:**
```json
{
  "year": 2025,
  "month": 6
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| year | int | Yes | Settlement year |
| month | int | Yes | Settlement month (1-12) |

**What happens internally:**
1. Finds all **Completed Offer Transactions** where `completedAt` falls in the given month
2. Finds all **Completed Partner Transactions** where `completedAt` falls in the given month
3. Groups both by `(providerType, providerId)`
4. For each provider:
   - **Offers:** Sums `monetaryValue` for `totalTransactionAmount` and `totalCommissionAmount`, sums `pointsDeducted` for `totalPointsDeducted`
   - **Partners:** Sums `transactionAmount`, `commissionAmount`, `pointsAwarded` as before
   - Creates or updates a `ProviderSettlement` record (Status = Pending)
   - If a provider has both offer and partner transactions, they combine into ONE settlement

**Response:** `200 OK` -- Returns the number of new settlements created (int)

**Errors:**
- `401` Unauthorized
- `403` Forbidden

## Settlement Integration

Offer and partner transactions are both included in the monthly settlement process:

```
Monthly Settlement Generation:
  1. Get all Completed OfferTransactions for the month
  2. Get all Completed PartnerTransactions for the month
  3. Group BOTH by (ProviderType, ProviderId)
  4. For each provider:
     OFFERS:
     - Sum MonetaryValue = What Cable pays the provider
     - Sum PointsDeducted = Total points taken from users
     PARTNERS:
     - Sum CommissionAmount = What Cable owes for partner transactions
     - Sum TransactionAmount = Total purchase volume
     - Sum PointsAwarded = Total loyalty points given to users
  5. Create ONE ProviderSettlement record per provider per month
  6. If provider already has a settlement (from offers), partner data ADDS to it
```

**Settlement Fields:**
- `totalPointsAwarded` -- from partner transactions (users earn points)
- `totalPointsDeducted` -- from offer transactions (users spend points)
- `totalCommissionAmount` -- combined: monetary value (offers) + commission (partners)

**Settlement Lifecycle:**
```
[Generated] -> Pending -> Invoiced -> Paid
                             -> Disputed
```

## Background Job: Expire Offer Transactions

A background job periodically runs to mark expired transactions:

- Finds all transactions where `Status = Initiated` and `CodeExpiresAt < now`
- Sets their `Status = Expired`
- This prevents users from scanning codes that have passed their expiry time

The scan endpoint also checks expiry in real-time -- if a user tries to scan an expired code, it gets marked as Expired and an error is returned.

## Business Rules Summary

1. **Offers cost points** -- Users spend loyalty points to redeem offers. Each offer has a fixed `pointsCost` (e.g. 100 points for a free coffee).

2. **Fixed monetary value** -- Each offer has a fixed `monetaryValue` that Cable pays the provider per redemption (set by admin/provider, e.g. 3.000 KWD).

3. **Offer approval required** -- Offers start as Pending. Only admin-approved offers can be used for transactions.

4. **Snapshot values** -- Points cost and monetary value are locked at transaction creation time. Admin changes to the offer only affect future transactions.

5. **Code expiry** -- Each code has a TTL set by the offer's `offerCodeExpiryMinutes`. Expired codes return an error and the transaction is auto-marked as `Expired`.

6. **Only Initiated codes can be scanned** -- Completed, Expired, and Cancelled codes cannot be scanned.

7. **Only the creating staff can cancel** -- The `ConfirmedByUserId` (staff who generated the code) must match the current user to cancel.

8. **Points deducted before completion** -- Points are deducted from the user's wallet BEFORE the transaction is marked as Completed. If the user has insufficient points, the deduction fails and the transaction stays as `Initiated` (can be retried or cancelled).

9. **No user ID at creation** -- When provider creates the transaction, `UserId` is null. It's set only when a user scans the code.

10. **Max uses per user** -- Checked at scan time (counts only Completed transactions). If a user has reached the limit, the scan is rejected.

11. **Max total uses** -- Checked at creation time. `CurrentTotalUses` increments when the provider creates a transaction, not when the user scans.

12. **Validity dates** -- `ValidFrom` and `ValidTo` are checked when the provider creates a transaction. The offer must be within its validity window.

13. **Bilingual support** -- Offers have English and Arabic fields (title, description).

14. **Combined settlements** -- If a provider has both offer and partner transactions, they are combined into one settlement per month. `TotalPointsAwarded` tracks partner points, `TotalPointsDeducted` tracks offer points.
